## Section 1: Signal Inventory

### 1.1 CoreLocation Signals

**GPS Speed (`CLLocation.speed`)**  
Instantaneous speed in m/s. Negative (-1) = invalid. Typical driving: >7–8 m/s. Apps receive this with every `CLLocation` update via `locationManager(_:didUpdateLocations:)`.

**GPS Course (`CLLocation.course` + `courseAccuracy`)**  
`course`: heading in degrees from due north (clockwise). `courseAccuracy` (iOS 13.4+): ±degree error. Valid course + consistent non-zero speed strongly indicates vehicular travel.

**Horizontal Accuracy (`CLLocation.horizontalAccuracy`)**  
Typically 5–15m for open-sky highway driving. Not injectable from outside.

**Significant Location Change (`startMonitoringSignificantLocationChanges`)**  
Fires on cell tower/Wi-Fi topology changes (~500m+ movement). Consistent with driving. Common background tracking mode for apps like Life360.

**`CLLocationSourceInformation.isSimulatedBySoftware` (iOS 15+)**  
Set by Core Location when location is generated by official software simulation (Xcode GPX debugger). Apple DTS engineer confirmed: only Xcode/Core Location simulation sets this flag. Third-party tools that operate as external location providers do not set it. DVT behavior on physical device: confirmed true (same `com.apple.dt.simulatelocation` service as Xcode).

### 1.2 CoreMotion Signals

**`CMMotionActivity.automotive`**  
`CMMotionActivityManager` classifies movement: `stationary`, `walking`, `running`, `cycling`, `automotive`, `unknown`. States are not mutually exclusive. Classification algorithm not documented. Community analysis suggests:
- Low speeds: magnetometer anomalies from metal vehicle enclosure
- Moderate/high speeds: accelerometer frequency/amplitude patterns (road vibration, suspension, powertrain)
- Runs on M-series coprocessor (M7+, iPhone 5s+), independently of main CPU
- `isActivityAvailable()` returns false on Xcode Simulator

**Raw Accelerometer/Gyroscope via `CMDeviceMotion`**  
`CMMotionManager.startDeviceMotionUpdates()` delivers raw IMU data. Apps doing own activity classification (Life360/Arity) consume this raw stream. Driving signatures: road vibration 4–20 Hz, hard braking (negative X decel), acceleration (positive X), turns (lateral Y + gyro Z rotation rate), speed bumps (Z impulse).

**Barometer (`CMAltimeter`)**  
Not directly used for driving detection but can confirm elevation changes consistent with road grade.

**Pedometer (`CMPedometer`)**  
Step count drops to zero during automotive travel. Combined with speed > 15 km/h: highly reliable driving indicator.

### 1.3 Connectivity and Context Signals

**Bluetooth HFP/A2DP Vehicle Profile**  
iOS Driving Focus activates when connected to vehicle BT using HFP. Apps can observe Bluetooth state with `CBCentralManager` (requires Bluetooth permission).

**Wi-Fi SSID changes**  
Rapid, continuous SSID changes correlate with driving through urban environments. Not commonly used as primary signal.

**Cell Tower Handoffs**  
Significant Location Change events have natural correlation with highway driving.

**CarPlay**  
Explicit CarPlay APIs in MediaPlayer/CarPlay frameworks. Strongest vehicle context signal.

## Section 3: CoreMotion Automotive — Hard No

1. `CMMotionActivityManager.isActivityAvailable()` returns `false` on Xcode Simulator.
2. On physical devices, M-series coprocessor firmware reads raw IMU. No DVT channel exposes it.
3. Full DVT instrument channel list: `CoreProfileSessionTap`, `Sysmontap`, `ProcessControl`, `DeviceInfo`, `ActivityTraceTap`, `NetworkMonitor`, `ApplicationListing`, `Screenshot`, `XCUITest`, `ConditionInducer`, `LocationSimulation`. No `MotionSimulation`, `ActivitySimulation`, or `IMUSensor` channel.
4. `ActivityTraceTap` is read-only telemetry — not an injection channel.
5. WWDC sessions on CoreMotion (2014–2023): no simulation or test injection pathway ever described.
6. GeoFake (GitHub: AlohaYos/GeoFake): achieves fake `CMMotionActivity` via method swizzling inside the app under test. In-process only. No effect on other apps.

**Conclusion:** No external, OS-level pathway to inject `automotive` activity state into an unmodified app on a physical device using documented tools.

## Section 4: GPX Timestamped Playback

**Xcode Simulator:** When `<time>` tags present, iOS derives speed from distance/elapsed time and injects into `CLLocation.speed`. However, multiple community sources report `CLLocation.speed` always returns -1 in the simulator — appears the speed derivation was removed after Xcode 9.

**DVT physical device (pymobiledevice3):** pymobiledevice3 issue #340 ("Simulate location with speed, heading and altitude") closed without implementation in 2022. DVT `simulate-location set` takes only `lat lon`. Whether GPX `play` path derives and populates `CLLocation.speed` on physical devices: **unconfirmed, likely no**.

**App behavioral differences:**
- Life360: speed threshold ~15 mph (6.7 m/s). If `CLLocation.speed = -1`, Life360 may compute approximate speed from successive coordinate deltas and timestamps.
- Google Maps: transitions to driving directions mode when speed consistently > ~10–15 km/h. Coordinate-only simulation at correct spacing can trigger this.
- Find My: uses significant location change monitoring, not continuous GPS. No specific "driving mode."

## Section 5: Apple Developer Tooling Scope

**Xcode Simulator (simctl):**
- `xcrun simctl location <device> set <lat> <lon>` and GPX file-based simulation
- `CMMotionActivity`: `isActivityAvailable()` = false. No activity updates fire. No simctl command for motion activity injection in any Xcode version through 16.x.
- Speed derivation: removed after Xcode 9 (community reports)
- Heading simulation: possible via private CoreSimulator API `startLocationSimulation(withDistance:speed:waypoints:)` on Simulator only. Not available on physical devices.

**DVT Protocol (pymobiledevice3, physical device):**
- `LocationSimulation` channel only: set lat/lon, play GPX, clear
- No `MotionSimulation` or speed/heading injection
- Feature request #340: explicitly confirmed limitation, closed without resolution

**XCTest / XCUITest:**  
Requires app under test to be launched by XCTest. Cannot inject into arbitrary third-party apps running standalone.

## Section 6: Third-Party App Detection

### Life360

- Data collected: GPS, accelerometer, gyroscope, compass, Bluetooth
- Permissions: Location (Always), Motion & Fitness, Background App Refresh
- Drive trigger: Speed > ~15 mph (6.7 m/s) sustained over ~0.8 km minimum
- Speeding detection: Speed > 80 mph for 30 seconds
- Driving analytics partner: Arity, LLC (Allstate subsidiary)
- Crash detection: accelerometer + GPS, vehicle speed > 25 mph
- Arity patent US11834051: "sequential micro-activity based driver detection" using IMU data patterns

**What simulation meets:** GPS speed threshold (coordinate injection at driving speed)  
**What simulation cannot meet:** CMMotionActivity automotive; accelerometer hard-braking/acceleration events; Arity IMU data

### Google Maps

- No documented algorithm. Behavioral observation: switches to driving navigation mode when moving > ~10 km/h.
- Mode switch appears primarily speed-based from `CLLocation.speed` or derived
- iOS CoreMotion: Google cannot customize CMMotionActivity at OS level

**What simulation achieves:** Coordinate injection at driving speeds likely triggers driving mode.

### Apple Driving Focus / Do Not Disturb While Driving

- Primary trigger: Bluetooth vehicle connection (HFP or A2DP) OR accelerometer-based vehicle motion detection
- Secondary: speed from CoreLocation
- Multi-sensor aggregate: accelerometer, magnetometer, gyroscope
- Passenger detection: user can tap "I'm Not Driving" — system is not certain enough to auto-confirm

**What simulation cannot achieve:** Stationary device receives coordinate updates; zero IMU activity.

---

## Section 7: Definitive Gaps

1. **`CMMotionActivity.automotive` injection into unmodified app on physical device** — no DVT channel, no public API.
2. **`CLLocation.speed` / `CLLocation.course` injection via DVT** — DVT accepts only lat/lon.
3. **Suppressing `isSimulatedBySoftware = true`** when using official Xcode/DVT simulation — cannot suppress.
4. **Bluetooth vehicle context / CarPlay** — requires real hardware.
5. **Raw IMU injection** on physical device externally — IMU reads real hardware only.
6. **Life360 Arity IMU events** (hard braking, acceleration scoring) — requires real accelerometer data.
